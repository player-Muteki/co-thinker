# 架构决策记录（ADR）

## 1. 说明

本文记录 Co-Thinker 当前阶段的重要架构决策。每条 ADR 都包含背景、决策、备选方案、影响和权衡。后续如果实现阶段发现假设不成立，应新增 ADR 或将原 ADR 标记为 Superseded，而不是直接删除历史决策。

## ADR-001：采用模块化单体架构

### 状态

Accepted

### 背景

Co-Thinker 当前目标是构建本地或轻量团队使用的 RAG 智能问答系统。核心能力包括文档导入、检索、生成、多轮对话和 Streamlit UI。项目仍处于 MVP 阶段，尚无多用户、高并发、跨服务独立扩展等需求。

### 决策

采用模块化单体架构：

- 一个 Streamlit 应用作为入口。
- 业务逻辑拆分为 `config`、`ingest`、`retriever`、`generator`、`chat_engine` 等模块。
- 本地目录保存文档、向量库、manifest 和对话历史。

### 备选方案

- **微服务架构**：将文档导入、检索、生成拆成独立服务。拒绝原因：早期运维复杂度高，收益不足。
- **前后端分离 + FastAPI**：更适合多人共享和 API 化。暂缓原因：MVP 优先验证 RAG 闭环。
- **纯脚本/Notebook**：开发快，但不利于长期维护和 UI 交互。

### 影响

正面：

- 部署简单。
- 调试容易。
- 模块边界仍然清晰，后续可演进服务化。

负面：

- 不适合高并发。
- 长耗时导入会阻塞单进程 UI。
- 多用户权限隔离能力弱。

### 权衡

当前优先选择快速落地和低运维成本，而不是提前为高并发做复杂架构。

## ADR-002：使用 Streamlit 构建 MVP UI

### 状态

Accepted

### 背景

项目需要可视化交互界面，包括文档上传、文档管理、聊天问答、设置和会话历史。MVP 需要快速构建可用界面，而不是投入大量前端工程。

### 决策

使用 Streamlit 作为 UI 框架。

### 备选方案

- **Gradio**：适合模型 demo，但复杂文档管理和多页面状态管理不如 Streamlit 灵活。
- **React + FastAPI**：体验和扩展性更好，但开发成本明显更高。
- **命令行界面**：实现简单，但不满足 README 的可视化交互界面目标。

### 影响

正面：

- 快速实现上传、聊天、表单和状态展示。
- Python 全栈，减少前后端切换。
- 适合本地知识库工具。

负面：

- Streamlit 重渲染机制需要谨慎管理状态。
- 不适合复杂权限、多人协作和高并发。

### 权衡

MVP 阶段优先交互闭环和开发速度；若未来需要团队协作和权限系统，再迁移到 FastAPI + Web 前端。

## ADR-003：使用 ChromaDB 作为本地向量库

### 状态

Superseded by ADR-011

### 背景

RAG 系统需要持久化 embedding 向量并支持相似度检索。当前目标是单机运行、少量到中等规模文档，优先降低部署成本。

### 决策

使用 ChromaDB 本地持久化目录作为向量库。

### 备选方案

- **FAISS**：性能好，但 metadata/filter/persistence 需要更多自建逻辑。
- **Qdrant**：功能强，适合服务化部署，但 MVP 需要额外服务或容器。
- **PostgreSQL + pgvector**：适合已有数据库和多用户场景，但当前引入数据库过重。

### 影响

正面：

- 免部署，适合本地开发。
- 与 LlamaIndex 集成方便。
- 支持 metadata 和持久化。

负面：

- 大规模数据和并发场景能力有限。
- 本地目录损坏时需要重建索引。

### 权衡

选择 ChromaDB 是为了 MVP 简单可靠；只要保留 `VectorStore` 封装边界，后续可迁移到 Qdrant 或 pgvector。

## ADR-004：采用 DeepSeek Chat 作为生成模型

### 状态

Accepted

### 背景

系统需要调用大模型生成最终答案，并要求能结合检索上下文、多轮历史和中文回答。DeepSeek 提供 OpenAI 兼容 API，便于与 LlamaIndex OpenAI 包装器集成。

### 决策

默认使用 DeepSeek Chat API 作为生成模型。

### 备选方案

- **OpenAI Chat Models**：生态成熟，但成本和用户偏好可能不匹配。
- **Claude API**：长文本和推理能力强，但当前项目计划中已偏向 DeepSeek。
- **本地 LLM**：隐私更好，但硬件、部署和质量成本高。

### 影响

正面：

- OpenAI 兼容接口，集成成本低。
- 中文能力和成本适合 MVP。
- 可支持流式输出。

负面：

- 依赖外部 API 和网络。
- 需要处理限流、超时和参数兼容问题。
- 敏感内容会发送到第三方模型服务，需提醒用户。

### 权衡

当前优先使用外部 API 保证质量和开发速度；若未来强调私有化，可替换为本地模型或企业模型服务。

## ADR-005：使用 OpenAI `text-embedding-3-small` 作为 embedding 模型

### 状态

Superseded by ADR-012

### 背景

文档和查询需要向量化。Embedding 模型直接影响检索质量和成本。MVP 需要稳定、通用、易集成的方案。

### 决策

默认使用 OpenAI `text-embedding-3-small` 作为 embedding 模型。

### 备选方案

- **本地 embedding 模型**：隐私和成本可控，但部署复杂，质量需评估。
- **其他云 embedding**：可能更便宜或区域友好，但需要额外适配。
- **仅 BM25 不使用 embedding**：无法满足语义检索目标。

### 影响

正面：

- 通用质量稳定。
- 成本相对可控。
- 与 LlamaIndex 集成成熟。

负面：

- 依赖外部 API。
- 文档内容会发送到第三方服务做 embedding。
- 需要控制重复导入，避免重复计费。

### 权衡

MVP 优先质量和稳定性；通过 manifest hash 去重控制成本，并保留替换 embedding 的工厂函数。

## ADR-006：默认采用混合检索（向量 + BM25 + RRF）

### 状态

Accepted

### 背景

Co-Thinker 需要支持自然语言知识问答，也需要支持代码文件、配置项、专有名词和文件路径查询。单一向量检索对精确符号不敏感；单一 BM25 又无法处理语义相似表达。

### 决策

默认使用混合检索：

1. 向量检索召回语义相关片段。
2. BM25 检索召回关键词和代码符号相关片段。
3. 使用 RRF 按排名融合结果。

### 备选方案

- **仅向量检索**：实现简单，但代码符号和专有名词可能漏召回。
- **仅 BM25**：成本低，但语义理解弱。
- **Cross-encoder reranker**：质量可能更好，但增加模型依赖和延迟。

### 影响

正面：

- 兼顾语义和关键词。
- RRF 不依赖不同检索器分数尺度一致。
- 适合文档 + 代码混合知识库。

负面：

- 实现复杂度高于单一路径。
- BM25 中文分词和代码 tokenization 需要后续调优。

### 权衡

MVP 先用 RRF 混合召回，后续如质量不足再增加 reranker，而不是一开始引入更重模型。

## ADR-007：使用本地 JSON 保存对话历史和文档清单

### 状态

Accepted

### 背景

系统需要保存对话历史、当前会话、文档导入状态、文件 hash 和错误信息。当前没有多用户并发写入需求。

### 决策

使用本地 JSON 文件：

- `storage/chat_history.json` 保存对话历史。
- `storage/document_manifest.json` 保存文档清单。

### 备选方案

- **SQLite**：更可靠，适合查询和并发，但初期开发成本稍高。
- **PostgreSQL**：适合服务化和多用户，但 MVP 过重。
- **仅内存存储**：实现简单，但重启丢失状态，不满足可用性目标。

### 影响

正面：

- 简单可读，便于调试。
- 无需额外服务。
- 适合本地单用户。

负面：

- 并发写入能力弱。
- 文件损坏时需要备份和恢复策略。
- 查询统计能力有限。

### 权衡

MVP 选择 JSON 降低复杂度，但实现原子写入和损坏备份；后续可迁移到 SQLite。

## ADR-008：无检索结果时不调用 LLM

### 状态

Accepted

### 背景

RAG 系统最大的质量风险之一是模型在没有知识库依据时编造答案。如果检索结果为空，继续调用 LLM 会增加成本，并提高幻觉风险。

### 决策

当检索结果为空时，`RAGGenerator` 直接返回固定的“知识库中未找到足够信息”提示，不调用 LLM。

### 备选方案

- **仍调用 LLM，让模型解释无结果**：语言更自然，但成本和幻觉风险更高。
- **回退为通用聊天**：用户体验可能更顺滑，但违背“基于知识库”的系统定位。

### 影响

正面：

- 降低幻觉风险。
- 降低 API 成本。
- 用户能明确知道知识库覆盖不足。

负面：

- 对某些常识性问题也会拒答。
- 如果检索器漏召回，用户会误以为知识库没有答案。

### 权衡

项目定位是领域知识库问答，因此优先保证可信和可追溯，而不是通用聊天能力。

## ADR-009：优先支持 md/txt/代码文件，增强阶段再支持 Office/PDF

### 状态

Accepted

### 背景

README 明确提到支持 md、txt、代码文件等格式批量导入。PDF、DOCX、PPTX 解析会引入更多依赖和失败模式，尤其扫描版 PDF 和复杂表格。

### 决策

MVP 优先支持：

- Markdown。
- 纯文本。
- 常见代码文件。

P1 再支持 PDF、DOCX、PPTX、CSV、JSON/YAML。

### 备选方案

- **一开始支持所有格式**：用户体验更完整，但解析复杂度和测试负担高。
- **只支持 Markdown**：实现最快，但不满足 README 的多格式目标。

### 影响

正面：

- 更快交付核心 RAG 闭环。
- 代码文件支持能覆盖项目知识库问答的重要场景。
- 降低早期解析失败风险。

负面：

- 早期不能覆盖所有常见办公文档。
- 用户可能需要先转换文档格式。

### 权衡

先满足 README 明确提到的 md/txt/代码文件，再逐步扩展复杂格式。

## ADR-010：API Key 只通过环境变量或运行时输入提供，不由 UI 持久化写入

### 状态

Accepted

### 背景

系统需要 DeepSeek 和 OpenAI API Key。UI 设置页可以提高易用性，但把密钥从 UI 写入 `.env` 或其他文件存在误提交和权限风险。

### 决策

MVP 中：

- 持久配置通过 `.env` 或系统环境变量提供。
- UI 设置页允许运行时输入 API Key，但只保存在当前进程/session 中。
- 不自动把 API Key 写入 `.env`。

### 备选方案

- **UI 写入 `.env`**：方便，但有误提交风险。
- **只允许 `.env`**：安全边界清晰，但用户体验稍差。
- **系统密钥管理器**：更安全，但实现复杂且跨平台成本高。

### 影响

正面：

- 降低密钥落盘和误提交风险。
- 用户仍可通过 UI 临时试用。
- 安全行为简单可解释。

负面：

- UI 中输入的 Key 重启后失效。
- 用户需要理解 `.env` 配置方式。

### 权衡

优先安全默认值；如未来需要持久化密钥，应引入明确的加密存储或系统 keyring。

## ADR-011：使用 JsonVectorStore 替代 ChromaDB

### 状态

Accepted

### 背景

实现阶段发现 ChromaDB 对 MVP 场景过重：需要额外安装本地库依赖（`chromadb`），增加了首次启动的失败率。MVP 的数据规模（< 100 个文档、< 10k chunks）下，全内存扫描的延迟可以接受。同时项目需要嵌入向量和文档清单的强一致性（同一文件更新时替换所有 chunks），ChromaDB 的事务边界不如自定义 JSON 存储直观。

### 决策

使用自定义 `JsonVectorStore` 替代 ChromaDB：

- 将所有 chunks（文本 + 元数据 + 向量）存储在一个 `vectorstore/chunks.json` 中。
- 使用原子写入（tmp + replace）保证写入不损坏。
- 检索时全量加载到内存，O(N) 扫描计算相似度。
- 保留 `JsonVectorStore` 的封装边界，未来可替换为专用向量库。

### 备选方案

- **ChromaDB**：功能完整，但增加了部署依赖和版本兼容风险。Superseded。
- **FAISS**：检索性能更好，但 metadata 过滤和增量更新需要额外实现。
- **SQLite + 向量扩展**：查询能力强，但 MVP 开发成本稍高。

### 影响

正面：

- 零额外依赖，`pip install` 即可运行。
- 原子写入保证数据一致性。
- 便于调试 —— 可直接读取 chunks.json。

负面：

- 全量 O(N) 扫描，10k+ chunks 后检索延迟可能超过 1 秒。
- 只适合单进程单用户场景。
- 向量嵌入以 JSON 存储，文件体积较大。

### 权衡

MVP 优先降低部署摩擦和保证数据一致性。当数据规模增长到需要索引加速时，将 JsonVectorStore 的内部实现替换为 FAISS 或迁移到 Qdrant，对外接口不变。

## ADR-012：移除 OpenAI Embedding，仅使用 DeepSeek

### 状态

Accepted

### 背景

MVP 阶段需要简化外部依赖。OpenAI Embedding API 需要独立的 API Key 和网络访问，增加了首次启动的配置门槛。DeepSeek API 不提供 embedding 模型接口。

### 决策

移除 OpenAI Embedding 依赖：

- 不再需要 `OPENAI_API_KEY`。
- 移除 `llama-index-embeddings-openai` 依赖。
- 向量检索回退到基于 token 重叠的相似度计算（不需要外部 API）。
- BM25 关键词检索作为主检索路径。
- 仅保留 DeepSeek Chat API 用于答案生成。

### 备选方案

- **保留 OpenAI Embedding**：检索质量更高，但增加配置门槛和外部依赖。
- **使用本地 embedding 模型**：隐私更好，但部署复杂度和质量难以平衡。
- **其他免费 embedding API**：可能不可靠或有使用限制。

### 影响

正面：

- 零外部 embedding 依赖，`DEEPSEEK_API_KEY` 即可运行全部功能。
- 减少 API 调用次数和成本。
- 降低首次启动的配置复杂度。

负面：

- 失去语义向量检索，检索质量下降（仅靠关键词 + token 重叠）。
- 对同义词、近义词的表达理解能力变弱。

### 权衡

MVP 阶段优先降低部署门槛和外部依赖。未来如果需要语义检索，可以通过 ADR-011 中定义的 VectorStore 接口接入本地 embedding 模型或第三方 embedding API，而不需要修改业务代码。
